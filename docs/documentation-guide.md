# 文档编写规范

## 核心原则

### 简洁优先
- 避免冗余，每句话都要有价值
- 用最少的文字传达核心信息
- 避免重复表述

### 设计导向
- 突出核心设计思路和架构决策
- 说明"是什么"和"为什么"
- 避免过度描述实现细节

### 无代码示例
- 文档中不包含具体实现代码
- 不包含 hello world 级别的示例
- 不粘贴配置文件或代码片段

### 概念清晰
- 用自然语言描述设计思路
- 说明接口定义和数据流（如果需要）
- 避免让读者猜测设计意图

### 格式规范
- **列表行**: 列表项开始尽量不要使用加粗（`**`），仅在需要强调时使用
- **标题层级**: 合理使用标题层级，避免过多嵌套

## 文档应包含的内容

- **设计目标和核心思路**: 说明要解决什么问题，为什么这样设计
- **架构概览和模块关系**: 描述整体结构和各部分的关系
- **关键决策的背景原因**: 解释为什么选择某个方案，考虑了哪些因素
- **接口定义和数据流**: 如果需要，描述关键接口和数据流向
- **开发流程和规范要点**: 概括性的指导原则

## 文档不应包含的内容

- **具体实现代码**: 实现细节应通过代码本身和注释表达
- **代码示例**: 代码示例容易过时，增加维护负担
- **详细的配置文件内容**: 配置应通过配置文件本身和注释说明
- **IDE 使用教程**: 不作为项目文档的一部分
- **第三方库的基础教程**: 可通过链接引用官方文档

## 文档组织原则

### 按职责分类
- **architecture/**: 系统架构和技术选型
- **modules/**: 功能模块的设计和接口定义
- **features/**: 具体功能的设计思路和实现说明
- **database/**: 数据库设计和安全要求
- **manual/**: 产品使用手册和操作指南

### 命名规范
- 使用 kebab-case 命名文件和目录
- 文件名应清晰反映内容主题
- 避免使用缩写和模糊的名称

## 文档维护

- 保持文档与代码同步更新
- 定期审查和优化过时内容
- 移除不再适用的文档
